= Cutover clients and clean up source SVM after ASA r2 migration.
:icons: font
:imagesdir: ../media/

[.lead]
After a storage virtual machine (VM) is migrated from an ASA cluster to an ASA r2 cluster, by default, clients are automatically cutover to the ASA r2 cluster and the storage VM on the ASA cluster is automatically removed. If you chose to disable automatic cutover and removal of the ASA storage VM during the migration, you need to perform these steps manual after the migration is complete.

== Manually cutover clients to an ASA r2 system after an SVM migration
If you disable automatic client cutover during the migration of a storage VM from an ASA cluster to an ASA r2 cluster, after the migration is successfully complete, perform the cutover manually so the ASA r2 storage VM can serve data to clients.

.Steps

. On the ASA r2 cluster, manually execute client cutover:
+
[source,cli]
----
vserver migrate cutover -vserver <SVM_name>
----

. Verify that the cutover operation is complete:
+
[source,cli]
----
vserver migrate show
----

.Result

Data is being served to your clients from the storage VM on your ASA r2 cluster.  

.Whatâ€™s next?
You are now ready to remove the storage VM from the ASA cluster.

== Manually remove an ASA SVM after migration to an ASA r2 cluster
If you disable automatic source cleanup during the migration of a storage VM from an ASA cluster to an ASA r2 cluster, after the migration is successfully complete, remove the SVM from the ASA cluster to free the storage space.

.Before you begin

Your clients should be serving data from the ASA r2 cluster.

.Steps

. From the ASA cluster, verify that the status of the ASA SVM is *Ready for source cleanup*:
+
[source,cli]
----
vserver migrate show
----

. Remove the ASA storage VM:
+
[source,cli]
----
vserver migrate source-cleanup -vserver <SVM_name>
----

.Result
The SVM on your ASA cluster is removed.
